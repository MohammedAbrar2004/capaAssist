# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A ground-up rebuild of the CAPA (Corrective and Preventive Action) AI Assist module for SoapBox.Cloud — **API service only, no frontend**. The previous implementation lives in `reference/` and is being rebuilt in `service/` with corrected architecture, a clean DB schema, and stricter modularity. This is intended as a production-grade service, not a prototype.

Two capabilities carried over from the reference build: **Draft CAPA Action** (generate/improve actions) and **Weak CAPA Detection** (evaluate + score action quality). Multi-agent system using context engineering (no fine-tuning) — the model doesn't know the org; relevant org context is retrieved and injected at runtime per agent call.

## Database

**The real SoapBox.Cloud CAPA module runs on Oracle (multi-tenant) — but we build against a PostgreSQL mirror of it, not the live Oracle DB.** The reference was built against an invented, simplified 12-table Postgres schema. The real production schema (~50 Oracle tables: `VARCHAR2`/`CLOB`/`NUMBER`/`RAW(16)`/`SDO_GEOMETRY`, `VARCHAR2(50)` string IDs, `TENANT_ID` everywhere) is dumped in `version2/schema.csv`; the triage of which tables matter + the full type/concept mapping live in `phases/phase1.md` — read that, don't re-derive from the CSV.

Architecture:
- **Do not touch the live Oracle DB during the build.** We recreate the *relevant subset* (core CAPA tables + masters + the 3 `capa_ai_*` tables) in **PostgreSQL `capa_assist`** as a faithful mirror — **real Oracle table/column names**, **VARCHAR string IDs** (not SERIAL), **TENANT_ID** scoping. Skip Tier-5 auth/session/billing/APEX plumbing.
- **All DB access goes through a repository/adapter layer** (`repositories/`): a `CapaRepository` interface, with `PostgresCapaRepository` now and `OracleCapaRepository` later. Raw SQL lives **only** inside the active repository impl — never in agents/retrieval. Swap to Oracle = add one class + set `DB_BACKEND=oracle`; the Postgres impl can then be deleted as one self-contained file. `config.DB_BACKEND` selects the impl.
- **Tenant-scoped** to one synthetic test tenant (`CAPA_TENANT_ID=TENANT_ACERTECH`); the reference's 40 CAPAs are re-mapped into the mirrored schema under that tenant. Reference concepts with no home (assets/departments/employees) are dropped.
- Recurrence uses native `CAPA.IS_RECURRING`/`RECURRENCE_COUNT`/`LINKED_CAPA_IDS`; effectiveness is derived from `CAPA_REVIEWS`/`CAPA_ACTIONS.VERIFIED_*`/`CAPA_CLOSURE` (no single `effectiveness_result` column); root-cause category is derived from `CAPA_CATEGORIES`/`RCA_METHOD_ID` (no native column).
- Stack: `psycopg2` active now; `oracledb` deferred (commented in `requirements.txt`) until the Oracle adapter is built.
- **Tables added after the Phase 1 triage** (not in the original ~50-table inventory, each justified + documented in the phase that added it): `MSTR_TENANT_EMP` (Phase 2 — employee/org-directory table, assumed schema since `schema.csv` doesn't have it; see `phases/phase2.md` decision 1) and `CAPA_AI_ACTION_TAXONOMY` (Phase 3 — Tier 4 AI-side, action_type × theme reference data for the Generator; see `phases/phase3.md` decision 2).

`reference/backend/seeds/` and `analysis.md`'s data-audit section describe the **old** toy Postgres model — useful for the CAPA domain content and intelligence concepts, not for table shapes. Map concepts onto the real Oracle schema (mirrored in Postgres); don't reproduce the reference DDL.

## Build Approach

Phased, with a checkpoint (test + debug) between every phase before moving to the next:

1. **Setup** — project/folder scaffolding, environment, tooling.
2. **Seed all data** — re-map the reference's CAPA records into the real Oracle schema under one synthetic test tenant, plus the vector store collections. See `phases/phase1.md` for the table triage and seed plan.
3. **Context Retrieval Agent** — built and hardened first, to a high standard, before any other agent. Per the reference architecture, every downstream agent's output quality is bounded by this agent's context quality, so it gets the most scrutiny.
4. **Individual agents, one at a time** — Generator → Evaluator/Scoring Engine → Improver → Explainability → Orchestrator/API layer, mirroring the dependency order in `reference/plan.md`, each with its own test checkpoint before the next agent starts.

## Non-Negotiable Constraints (carried over from the reference design)

- **No fine-tuning.** Context engineering only.
- **LLM never scores.** Scoring is deterministic Python; LLMs reason semantically, a rule engine computes the score.
- **Generation and improvement are separate agents.**
- **Human-in-the-loop.** The AI never submits, approves, or closes a CAPA.
- **Inferred fields are always tagged** with `source: "inferred"` + a confidence score — no silent inference.
- **All agent inputs/outputs validate against Pydantic schemas.** No raw dicts crossing agent boundaries — this was violated in the reference build (`ContextPackage.effective_actions`/`relevant_sops`/`regulatory_context`/`enterprise_context` were raw dicts) and is called out explicitly in `analysis.md` as a real source of silent bugs. Fix it from the start in this rebuild.
- **Modularity / separation of concerns.** Every concern (schemas, retrieval, agents, scoring, API) lives in its own module — this was already the reference's intent but was not followed strictly everywhere; enforce it harder this time.

## Phase Docs (mandatory process)

Every phase gets exactly one file: `phases/phaseN.md`. Before building a phase, write the plan into that file (goal, decisions made, structure/approach, what's deferred). While/after building, update the same file in place — never create a second doc — until it covers: the idea, the implementation plan, what it does, how it does it, how to run it, how to test it. That file is the single source of truth for that phase; nothing decided about a phase should live only in chat history. Read the relevant `phases/phaseN.md` before resuming or extending work on that phase.
