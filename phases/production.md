# Production Readiness — what must change before this ships

Living checklist. Every item is something this codebase currently does for
*dev convenience* that is wrong, missing, or unsafe in production. Update as
phases progress — don't let this go stale.

---

## Database

- **`DB_BACKEND=oracle` adapter doesn't exist.** `config.DB_BACKEND` switch is wired, but `repositories/oracle.py` was never written — only `PostgresCapaRepository` exists. The whole point of the repository layer is this swap is supposed to be "add one class" — that class doesn't exist yet. Must be built, tested against real Oracle, before any production cutover. `oracledb` is commented out in `requirements.txt` — uncomment when building it.
- **`capa_ai_readonly` password is a weak, hardcoded literal** in `seeds/seed_readonly_role.sql` (tracked in git, not `.env`) — `capa_ai_readonly_pw`. Must be rotated to a real generated secret and pulled from a secret manager (not committed SQL) before prod. Same for the main `postgres` user's password (`1234` in `.env` — `.env` is gitignored so not leaked via git, but it's a throwaway dev password and must not reach any shared/staging DB).
- **`DB_PORT=5433`, `localhost` everywhere** — dev-only port (avoiding the default 5432 collision locally). Production connection string/host/port must come from real infra config, not these defaults.
- **Single synthetic tenant tested.** Everything (seed data, tests, this entire phase) runs against one `CAPA_TENANT_ID=TENANT_ACERTECH`. The schema is multi-tenant (`TENANT_ID` on every domain table) but multi-tenant *behavior* — query isolation under concurrent different-tenant load, no cross-tenant leakage — has never been exercised with more than one tenant.
- **`DB_POOL_MIN=2` / `DB_POOL_MAX=10`** are dev-scale guesses. Needs real sizing against expected concurrent request volume before prod.
- **NL2SQL's `statement_timeout = 5000`ms** is a guess, not load-tested. NL2SQL whitelist + schema description (`retrieval/nl2sql.py:_SCHEMA_DESCRIPTION`) is a **hand-maintained snapshot** — if `schema.sql` changes, this constant must be updated manually or NL2SQL silently degrades to guard-rejections / hallucinated columns. No introspection-based sync exists.

## LLM / `services/llm.py` + `config.py`

- **Ollama fallback tier assumes a local `ollama` binary in PATH** (`subprocess.Popen(["ollama", "serve"], ...)`). Not true in any container/prod environment by default. Before prod: either (a) bake an Ollama sidecar/image with the fallback models pulled, or (b) deliberately drop this tier in prod config and accept Groq+OpenRouter as the full fallback chain (cleaner — recommend this unless there's a hard requirement for a fully-offline last resort).
- **Groq/OpenRouter API keys are dev keys living in `.env`** — gitignored, so not leaked, but still raw keys on disk. Production needs these in a secret manager (Vault/AWS Secrets Manager/etc.), injected as env vars at deploy time, not a checked-out `.env` file.
- **`LLM_DEBUG_LOGGING`** defaults to `false` — good — but verify nothing flips it on in a prod env file by copy-paste-from-dev. If it's ever `true` in prod, full prompts/responses (potentially containing CAPA incident details — injury descriptions, names) hit stderr logs unredacted.
- **No rate limiting / cost ceiling** on LLM calls from this service's side — a retry storm or runaway client could burn through all 4 Groq keys + 4 OpenRouter keys + hit Ollama in a tight loop. Needs a circuit breaker / per-tenant request budget before prod traffic.
- **`call_llm`'s key-rotation cursor (`_groq_key_cursor`/`_or_key_cursor`) is in-process global state.** Fine for a single process; if this ever runs as multiple replicas behind a load balancer, each replica rotates independently — not a correctness bug, just means "all 4 keys exhausted" detection is per-replica, not global. Acceptable for now, worth knowing.
- **Generator now makes 2 LLM calls per `/capa/generate` request** (skeleton + enrich, Phase 3 Sub-Phase 3b), each with its own retry-once — worst case 4 calls for one HTTP request. Compounds the "no rate limiting / cost ceiling" item above; a cost ceiling needs to account for this multiplier, not just per-request count.
- **Audit trail stores the `CAPARecord` input + final response, not the actual rendered prompt(s) sent to the LLM.** `model_version` (`"<model>@<8-char hash of prompt file(s)>"`) tells you *that* a prompt changed, but reproducing the exact prompt text 6+ months later means finding the git commit whose `prompts/generator/*.jinja2` content hashes to that prefix — works only if git history is retained. Considered storing the full rendered prompt in `capa_ai_audit_trail` (fully self-contained) but deferred: it would duplicate data already in `capa_ai_context_packages` and meaningfully bloat audit rows, especially now that there are 2 prompts (skeleton + enrich) per generate call. Revisit if exact prompt reproducibility becomes a real compliance requirement, not just a nice-to-have.
- **Prompt templates must source the org name from `config.ORG_NAME`, never hardcode it.** Fixed for the Generator's templates (`prompts/generator/*.jinja2`) in Sub-Phase 3b, the Evaluator's templates (`prompts/evaluator/*.jinja2`) in Sub-Phase 4b, the Improver's template (`prompts/improver/rewrite.jinja2`) in Phase 5, and the Explainability template (`prompts/explainability/explain.jinja2`) in Phase 6 — all five agent prompt folders now use `{{ org_name }}` instead of a literal `"AcerTech Industries"`, verified via `grep -ril "acertech" prompts/`. Fully audited across all 5 agents as of Phase 6 — no outstanding gap here.

## NL2SQL (`retrieval/nl2sql.py`)

- **New runtime SQL-execution surface** — guarded (whitelist + tenant filter + keyword/comment/multi-statement rejection, tested against 10 attack patterns) but it is LLM-generated SQL running against a live DB role. This is the highest-risk piece of code added in Phase 2. Before prod: independent security review of the guard logic specifically (not just functional tests), and confirm `capa_ai_readonly`'s grants really are SELECT-only on exactly the whitelist (re-run `seeds/seed_readonly_role.sql`'s grant list as an audit, don't trust memory).
- **No query-cost guard beyond `statement_timeout`.** A guarded-but-expensive join (e.g. cross join across all whitelisted tables) could still be slow. Consider `EXPLAIN`-based cost checks or a stricter whitelist of allowed JOIN patterns if this becomes a real bottleneck.

## Vector store (ChromaDB)

- **`data/chroma/` is a local filesystem-persisted client** (`chromadb.PersistentClient`). Fine for a single dev machine; not appropriate for a multi-replica production deployment (no shared storage, no HA). Needs either a real Chroma server deployment, a managed vector DB, or a shared persistent volume strategy before scaling past one instance.
- **No incremental embedding pipeline for new CAPAs/actions.** `seeds/seed_vector_db.py` only ever ran once at seed time. Nothing in the runtime codebase inserts a newly created/closed/verified CAPA action into `historical_capas` — the vector store will silently go stale the moment real CAPAs start flowing through endpoints that don't exist yet (Phase 5). **Decided:** a standalone embedding-sync service, built **after the last build phase** (i.e. after Phase 5/Orchestrator+API), that scans for CAPAs/actions not yet present in `historical_capas` and embeds+upserts them — same shape as `seed_historical_capas()` but incremental/idempotent instead of one-shot. Not built yet. See `phases/phase2.md`'s vector-store section for the cross-reference.
- **fastembed is CPU-bound, in-process, lazy-loaded** (`config._FastEmbedEmbeddingFunction`). Fine at seed-time low QPS; under production query load, embedding every NL2SQL/vector-search call in-process could become a latency/throughput bottleneck. Consider a dedicated embedding service or batched/cached query embeddings if this matters at scale.

## Evaluator → Improver/Explainability eval persistence (`services/eval_cache.py`, Phase 5-6)

- **Explainability cannot run standalone — it hard-depends on a prior or fresh Evaluator run, same as Improver.** `agents/explainability.py::run()` takes `EvaluationResult`/`ScoringResult` as required args and never calls the Evaluator itself (same separation as Improver). `POST /capa/explain` resolves them via `_resolve_evaluation()` — cache hit (a prior `/capa/evaluate` or `/capa/improve` call in this process) or, on a miss, a fresh `evaluator.run()` (full 3-layer eval, 3 LLM calls) before the Explainability call can happen at all. There is no path to get an explanation without an evaluation existing first — by design (explaining *why* something scored low requires the score to already exist), but it means `/capa/explain` alone, cold-cache, costs 3 extra LLM calls it looks like it shouldn't need. Same multi-replica cache-miss cost problem as Improver below — not a new gap, just a second consumer of the same one.
- **`services/eval_cache.py` is in-process only — not a substitute for a real cache or for the durable record.** Two distinct persistence stories exist for the Evaluator's output, and they're easy to conflate:
  - **`action_id` path**: `/capa/evaluate` calls `repo.write_evaluation()` → a durable row in `capa_ai_evaluations` regardless of the in-memory cache. This is the real source of truth for that path.
  - **Ad-hoc `actions[0]` path** (no `action_id`): never persisted anywhere by design (no FK target — Phase 4 decision 1). The in-memory `eval_cache` is the *only* place this evaluation exists, and only for the lifetime of the process that computed it.
  - In a single-process dev run, `/capa/improve`'s cache-miss fallback (re-run `evaluator.run()` fresh) papers over both cases correctly. **In a multi-replica production deployment, a cache miss is the common case** (the `/evaluate` call and the later `/improve` call can land on different replicas) — the ad-hoc path then always pays for a second full Evaluator run (3 LLM calls) it shouldn't need to, and the `action_id` path *could* avoid that by reading `capa_ai_evaluations` back instead of re-evaluating, but currently doesn't.
  - **Why the `action_id` path isn't already wired to read that table back**: `capa_ai_evaluations` stores `score`/`weakness_level`/`dimension_results` (→ reconstructable as `EvaluationResult`) but **not** `RecurrenceResult` — recurrence was never added as a persisted column/JSON field. Reconstructing `EvaluationResult` from the DB row while silently defaulting `RecurrenceResult` to "no recurrence" would be worse than just re-evaluating — it would drop a real recurrence warning without any signal that it happened. Decided not to ship that half-fix.
  - **Real fix before prod, two options**: (a) replace `eval_cache`'s in-process `OrderedDict` with a shared cache (Redis) so cache hits work across replicas for both paths — fixes the common case without a schema change, but the ad-hoc path still loses its only copy on a true cold cache (eviction or restart); or (b) add a `recurrence_result JSON` column to `capa_ai_evaluations` (action_id path only) and read it back on `/improve`'s cache miss — fixes that path durably, doesn't help the ad-hoc path at all (still has nowhere to persist to). (a) is cheaper and covers more cases; (b) is more correct for the one path it covers. Not built — call this out explicitly if `/improve` or `/explain` is going to run as >1 replica before either lands.

## API layer (`main.py` / `api/routes.py`)

- **No authentication on any endpoint.** Routes as of Phase 7: `/health`, `POST /capa/context`, `POST /capa/generate`, `POST /capa/evaluate`, `POST /capa/improve`, `POST /capa/explain`, `GET /capa/{capa_id}`, `GET /capas`, `GET /capa/{capa_id}/audit` — none guarded. The 3 new Phase 7 GET routes take `tenant_id` as a plain query param specifically *because* there's no auth/session to infer it from (see `phases/phase7.md` decision 4) — i.e. this gap got wider, not just older, this phase. This is explicitly deferred (per `reference/CLAUDE.md`'s own Phase 6 notes: "authentication on any endpoint... explicitly deferred"). Must be built before any endpoint touching real CAPA data goes live — this is a hard blocker, not a nice-to-have.
- **`POST /capa/generate` is a new unauthenticated LLM-call surface with no per-request size/cost ceiling.** Anyone who can reach it can trigger a Groq/OpenRouter call (and, on quota exhaustion, a local Ollama cold-start) per request — same general LLM cost-ceiling gap noted above, now with a concrete route attached to it.
- **No CORS configuration.** Whatever frontend eventually calls this API will need explicit CORS origins set — currently unset.
- **No rate limiting / request validation beyond Pydantic schema validation.**

## Logging / observability

- **Single stderr logger, no log aggregation, no correlation/request IDs threaded through.** `capa_ai_audit_trail` captures request_id per AI call at the DB level (good), but application logs (`logging_config.py`) don't currently tag lines with that same request_id, making cross-referencing a specific request's logs with its audit trail row manual.
- **No metrics/tracing** (latency per agent call, LLM tier-fallback rate, NL2SQL guard-rejection rate, etc.) — would materially help operating the 3-tier LLM fallback and the NL2SQL guard in production; neither exists today.

## Testing

- **Integration tests self-skip (not fail) if the dev DB is unreachable.** Correct for local dev, but means CI must have a real reachable `capa_assist` instance or these 23 tests silently contribute nothing — confirm CI config actually provisions the DB before relying on "tests pass" as a release gate.
- **All LLM-touching tests hit the real Groq API** (`test_context_retrieval.py`, the enrich()/nl2sql smoke tests) — burns real quota on every CI run and is flaky under provider outages/rate limits. Before this matters at scale: build a mocked-LLM test tier (per question 3 above) so CI doesn't depend on live Groq availability or cost.
- **Found during Phase 7's checkpoint, not fixed (local env issue, not a code bug):** in this dev machine's `capa-ai` conda env, the `SSL_CERT_FILE` activation variable points at `<env>/ssl/cacert.pem`, which doesn't exist — the env's `ssl/` directory is missing entirely, even though `certifi` is installed with a valid `cacert.pem`. Every test that makes a live `httpx` call to Groq/OpenRouter (all 4 agents' integration suites) fails with `FileNotFoundError` building the SSL context. Confirmed environment-only by reproducing the identical failure on pre-Phase-7 tests (`test_explainability.py`, `test_generator.py`) — not something this build's code controls. Fix is local (repair/repoint `SSL_CERT_FILE`, or unset it so Python falls back to `certifi`), not a repo change; worth calling out if CI ever provisions environments the same way this one was provisioned.

## General / process

- **`reference/` directory** (prior implementation) should not ship in any production artifact/image — it's read-only context for this rebuild, not runtime code, but nothing currently prevents it from being copied into a build.
- **`schema.sql`'s `CREATE TABLE IF NOT EXISTS` idempotency was never tested against a real migration tool** (Alembic/Flyway/etc.) — fine for one-shot dev setup, not a substitute for versioned migrations once this has live production data that must evolve safely.
